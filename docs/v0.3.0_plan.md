# v0.3.0 Plan

This document outlines the next development milestone after the v0.2.0 CLIP vs PLIP release.

The goal of v0.3.0 is to add an experiment summary utility that turns saved evaluation outputs into a concise, human-readable Markdown report. This keeps the project focused on low-compute, reproducible evaluation rather than adding new model dependencies too quickly.

## Current Status

- CLIP and PLIP are supported through the same CLIP-style wrapper interface.
- Retrieval, zero-shot classification, and prompt sensitivity workflows can save structured reports.
- Full-test MHIST CLIP vs PLIP observations are documented.
- Prediction-collapse warnings and class distributions are available in saved zero-shot metrics.
- A first zero-shot Markdown summary utility is available through `pathvlm-litebench summarize-report --task zero-shot`.
- Outputs are generated locally under `outputs/` and should not be committed.

The next gap is broader result synthesis: users can summarize one zero-shot report, but retrieval reports, prompt-sensitivity reports, and multi-run comparisons still require manual reading of several JSON and CSV files.

## Milestone Goal

v0.3.0 should add a lightweight utility that reads saved report artifacts and writes an `experiment_summary.md` file.

The first version should focus on summarizing existing outputs, not running new experiments.

Expected output:

```text
outputs/experiment_summary.md
```

The summary should help users answer:

- What runs were compared?
- Which models and prompts were used?
- What were the key metrics?
- Did any run show prediction collapse or strong class bias?
- Which result should be interpreted cautiously?
- Where are the source report files?

## Why This Matters

PathVLM-LiteBench is positioned as a low-compute evaluation toolkit. A summary utility strengthens that positioning because it turns local experiment artifacts into a readable research note without requiring dashboards, databases, hosted services, or large dependencies.

This shows the project supports not only model inference but also reproducible reporting and failure-mode analysis.

## Inputs

The utility should support these existing report files:

- zero-shot `metrics.json`
- zero-shot `predictions.csv`
- zero-shot `errors.csv`
- retrieval `retrieval_metrics.json`
- retrieval `retrieval_results.csv`
- prompt sensitivity `prompt_sensitivity_metrics.json`
- prompt sensitivity `prompt_sensitivity_summary.csv`
- prompt sensitivity `prompt_sensitivity_details.csv`

The first implementation can make most inputs optional. If a file is missing, the summary should skip that section and record that the artifact was not provided.

## Outputs

The initial `experiment_summary.md` should include:

- title and generation timestamp
- source file list
- run metadata when available
- compact zero-shot classification table
- true and predicted label distributions
- confusion matrix summary
- prediction-collapse warnings
- retrieval Recall@K summary when available
- prompt sensitivity summary when available
- short limitations section

The output should be plain Markdown so it can be opened directly on GitHub.

## Non-Goals

v0.3.0 should not add:

- dashboards
- databases
- MLflow, W&B, or hosted tracking integrations
- automatic dataset upload
- real pathology images in reports
- model training or fine-tuning
- new model wrappers
- full WSI processing

The summary utility should only read local artifacts and write local Markdown.

## Current API

The first implemented utility functions are:

```python
build_zero_shot_experiment_summary(report_dir: str | Path) -> str
save_zero_shot_experiment_summary(
    report_dir: str | Path,
    output_path: str | Path | None = None,
) -> str
```

The implementation:

- parse known JSON report schemas with structured logic
- parse CSV artifacts only when needed
- tolerate missing optional fields
- avoid loading images
- avoid requiring model packages
- keep output deterministic

## Current CLI

The current CLI supports zero-shot report summaries:

```bash
pathvlm-litebench summarize-report \
  --task zero-shot \
  --report_dir outputs/zero_shot_demo
```

The CLI should not run models or download anything.

## Implementation Sequence

1. Add a small report-summary module. Done.
2. Implement Markdown rendering for zero-shot `metrics.json`, `predictions.csv`, and `errors.csv`. Done.
3. Add unit tests with tiny synthetic JSON/CSV fixtures. Done.
4. Add CLI subcommand. Done.
5. Add optional retrieval metrics support.
6. Add optional prompt sensitivity metrics support.
7. Document usage in `docs/real_patch_workflow.md`.
8. Add an example command using local `outputs/` paths without committing generated outputs.

## Testing Strategy

Tests should remain lightweight and should not depend on model downloads or real pathology data.

Recommended tests:

- zero-shot metrics JSON renders expected Markdown sections
- prediction-collapse warning is included when present
- missing optional sections are skipped cleanly
- CLI argument parsing writes the expected output file
- later: multiple report files produce a comparison table

Use small synthetic fixtures under `tests/` rather than real MHIST outputs.

## Documentation Updates

If implemented, update:

- `README.md`
- `docs/real_patch_workflow.md`
- `docs/project_positioning.md`
- release notes for the next version

Documentation should explain that generated summaries are local outputs and should not be committed when they contain dataset-specific results.

## Release Criteria

v0.3.0 should only be released when:

- existing CLIP and PLIP workflows still pass tests
- CI remains lightweight
- summary generation does not require model imports
- generated summaries do not copy images or embed real pathology data
- docs present the feature as experiment reporting, not benchmark certification
- release notes clearly describe supported input report types
