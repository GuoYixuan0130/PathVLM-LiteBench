# v0.9.0 Plan

This document outlines the next development milestone after the v0.8.0 prompt-scored patch-coordinate heatmap release.

v0.9.0 should improve reproducibility and comparison around prompt-scored patch-coordinate heatmaps. The goal is to make repeated prompt experiments easier to audit, compare, and rerun while keeping PathVLM-LiteBench patch-level, local-first, and lightweight.

## Current Status

- v0.8.0 is tagged and published as a GitHub Release.
- Main is now on development version `0.9.0.dev0`.
- v0.8.0 contains both coordinate heatmap paths:
  - `render-coordinate-heatmap` for existing score CSV artifacts
  - `score-coordinate-heatmap` for model-backed scoring from a coordinate manifest plus one text prompt
- Prompt-scored heatmap runs can be driven by CLI arguments or `patch_coordinate_heatmap_scoring` JSON config.
- Prompt-scored heatmap runs now save a metadata sidecar and support `--dry-run` validation without model loading.
- Saved prompt-scored score CSV artifacts can now be compared without model loading.
- CI tests scoring logic with fake models and does not download model weights.
- CONCH remains optional and gated.

## Milestone Goal

v0.9.0 should answer:

- Can users compare multiple prompt-scored patch-coordinate heatmaps without rerunning model inference?
- Can scored heatmap runs save enough metadata to make outputs easier to audit?
- Can users validate scoring inputs and output paths before loading a model?
- Can the workflow remain patch-level and local-first while supporting repeated prompt experiments?

## Preferred Scope

Recommended v0.9.0 scope:

1. Add run metadata output for prompt-scored coordinate heatmaps.
2. Add an artifact-only comparison utility for multiple score CSVs or heatmap runs.
3. Add a dry-run or validation mode for `score-coordinate-heatmap` that does not load models.
4. Add documentation for repeated prompt-scored coordinate heatmap experiments.
5. Keep CI coverage model-free by using fake model wrappers and saved artifacts.
6. Keep generated score CSVs, heatmaps, metadata files, reports, caches, and datasets ignored.

## Non-Goals

v0.9.0 should not add:

- whole-slide image file reading
- slide tiling or tissue detection
- slide pyramid rendering
- clinical diagnostic claims
- tumor boundary identification claims
- required model downloads in CI
- required CONCH access
- Hugging Face token handling
- committed generated heatmaps, score CSVs, metadata outputs, reports, caches, model weights, or real pathology images

## Candidate Work Items

Completed first slice:

- add a small metadata sidecar for `score-coordinate-heatmap` runs
- include prompt, model, device, manifest path, patch count, output paths, and timestamp-like run fields where appropriate
- add `score-coordinate-heatmap --dry-run` validation that resolves config and manifest metadata without loading a model
- add tests that verify dry-run behavior does not call model creation
- document dry-run and metadata behavior

Completed second slice:

- add an artifact-only comparison command for multiple prompt-scored coordinate score CSVs
- support user-provided run names for comparison outputs
- save a compact Markdown or CSV comparison summary under `outputs/`
- document how to compare repeated prompt-scored heatmap runs from saved artifacts

Remaining lower priority:

- add multi-prompt scoring over the same coordinate manifest
- add cached image embeddings for repeated prompt scoring
- add comparison utilities for multiple models over the same coordinate manifest

## Design Constraints

- Treat heatmaps as patch-coordinate score visualizations only.
- Keep `render-coordinate-heatmap` artifact-only and model-free.
- Keep any new comparison utility artifact-only unless explicitly scoped otherwise.
- Use existing model registry behavior and `--device auto` for model-backed scoring.
- Keep config fields explicit and serializable.
- Keep tests lightweight and independent of real pathology images.
- Keep public wording focused on patch-level evaluation and visualization.

## Release Criteria

v0.9.0 should only be released when:

- metadata or dry-run behavior is covered by local tests
- any comparison utility reads saved artifacts without model downloads
- docs explain output layout, model-loading behavior, artifact-only comparison behavior, and limitations
- README release links are updated during release preparation
- CI passes on the release commit and tag
