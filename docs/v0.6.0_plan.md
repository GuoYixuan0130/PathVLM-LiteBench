# v0.6.0 Plan

This document outlines the recommended next milestone after the v0.5.0 public patch benchmark workflow release.

The recommended v0.6.0 direction is a documented real public patch benchmark observation using the workflow introduced in v0.5.0. The goal is to move from workflow packaging to one reproducible, clearly scoped benchmark result without turning the project into a dataset-hosting or training repository.

## Current Status

- v0.5.0 is tagged and published as a GitHub Release.
- The project has documented public patch benchmark and prompt-grid workflows.
- Example configs are validated in offline tests.
- `validate-config` can check committed configs without loading models or datasets.
- CLIP, PLIP, and optional CONCH are implemented in the model registry.
- Generated datasets, outputs, demo patches, model weights, embeddings, and Python build metadata are ignored.
- A first MHIST sampled reproducibility audit has verified the v0.5.0 workflow for CLIP and PLIP.
- Sampled MHIST CLIP and PLIP zero-shot baseline configs are committed.
- `run-zero-shot-grid` supports runtime output path overrides for non-overwriting local audit runs.
- A sampled MHIST CLIP/PLIP/CONCH prompt-grid audit has completed successfully.

## Milestone Goal

v0.6.0 should answer:

- Can a new user reproduce one public patch benchmark observation from documented commands?
- Are the output directories, report names, run labels, and comparison summaries consistent enough for repeated local runs?
- Can the docs explain observed model behavior without clinical claims?
- Can optional CONCH access remain clearly separated from baseline CLIP/PLIP workflows?

## Preferred Scope

Recommended v0.6.0 scope:

1. Run one local public patch benchmark observation using the v0.5.0 workflow.
2. Document the exact commands, config choices, hardware notes, and artifacts produced.
3. Add a conservative observation document for CLIP and PLIP; include CONCH only if local gated access and runtime are stable.
4. Tighten output naming conventions if the real run reveals confusing paths or labels.
5. Add tests for any config or report-format behavior changed during the benchmark workflow.

## Non-Goals

v0.6.0 should not add:

- bundled datasets
- committed generated reports or predictions
- clinical diagnostic claims
- automatic Hugging Face token handling
- required CONCH downloads
- model training or fine-tuning
- full WSI processing as a required workflow

The optional WSI-level text-guided heatmap idea should remain lower priority unless the patch benchmark workflow is already easy to reproduce.

## Candidate Work Items

Highest priority:

- document one real public patch benchmark observation. Started with [v0.6.0_mhist_reproducibility_audit.md](v0.6.0_mhist_reproducibility_audit.md).
- verify public benchmark commands from a clean local output directory. Done for CLIP and PLIP.
- add committed benchmark configs for CLIP and PLIP sampled MHIST zero-shot baselines. Done.
- clarify output naming and comparison report conventions. Started with `run-zero-shot-grid --output-root`.
- run a sampled MHIST prompt-grid audit with CLIP, PLIP, and optional CONCH. Done.
- add a small release-facing audit checklist for benchmark reproducibility

Medium priority:

- add an example comparison report template that does not contain generated results
- add optional config fields for clearer run labels if needed
- document when to prefer CLIP-only, CLIP plus PLIP, or CLIP plus PLIP plus CONCH runs

Lower priority:

- explore a lightweight WSI heatmap prototype
- add additional prompt variants after the baseline benchmark observation is stable
- evaluate whether a PyPI release is useful for this project

## Release Criteria

v0.6.0 should only be released when:

- at least one real public patch benchmark observation is documented
- reproduction commands are clear enough to rerun locally
- generated outputs remain ignored and uncommitted
- CI remains offline and lightweight
- any new config or report behavior is covered by tests
- documentation frames results as local model-behavior observations, not clinical interpretation

## Suggested First Step

The next implementation step should be a benchmark reproducibility audit: run the documented v0.5.0 workflow on a local public patch dataset sample, then write down any command, config, or naming friction before changing code.
