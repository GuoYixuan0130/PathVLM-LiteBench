# v0.2.0 Plan

This document outlines a conservative plan for the next development milestone after the v0.1.0 CLIP baseline release.

The main goal for v0.2.0 is to prepare and, if feasible, add a pathology-specific VLM wrapper without breaking the low-compute, reproducible CLIP baseline workflow.

## Milestone Goal

v0.2.0 should extend PathVLM-LiteBench from a general CLIP baseline toolkit toward pathology-specific VLM comparison.

The preferred direction is:

1. Preserve the existing CLIP baseline pipeline.
2. Add a carefully scoped PLIP wrapper if the dependency and model-loading path is stable.
3. Compare CLIP and PLIP under the same retrieval, zero-shot, and reporting interfaces.
4. Keep tests lightweight and avoid model downloads in CI.

CONCH should remain a later milestone unless its access, dependency, and loading requirements are clear.

## Why PLIP Next

The MHIST CLIP baseline observation showed strong class bias and low `SSA` recall. This supports the next research question:

Can a pathology-specific VLM improve behavior on fine-grained pathology patch tasks under the same low-compute evaluation protocol?

PLIP is a natural next candidate because it is pathology-oriented and can potentially fit the current CLIP-style wrapper pattern.

## Non-Goals for v0.2.0

v0.2.0 should not attempt to add:

- full WSI processing
- high-throughput deployment
- large-scale training or fine-tuning
- multi-GPU support
- clinical diagnostic claims
- CONCH support unless the integration path is low-risk
- heavyweight dependencies such as OpenSlide or TIAToolbox in the core package

## PLIP Readiness Checklist

For the local-only validation protocol, see `docs/plip_feasibility_check.md`.

Before implementing PLIP, verify:

- [ ] exact Hugging Face model identifier
- [ ] model architecture compatibility with `transformers`
- [ ] processor/tokenizer class compatibility
- [ ] expected image preprocessing size and normalization
- [ ] text embedding API
- [ ] image embedding API
- [ ] output embedding normalization requirement
- [ ] CPU loading behavior
- [ ] CUDA loading behavior on laptop GPU
- [ ] model download size
- [ ] license and usage constraints
- [ ] whether authentication is required

If any of these are unclear, keep PLIP as a planned wrapper and document the blocker rather than merging a fragile implementation.

## Wrapper Design Principles

The PLIP wrapper should match the existing CLIP wrapper interface:

```python
encode_text(texts: list[str]) -> torch.Tensor
encode_images(images: list[Image.Image]) -> torch.Tensor
compute_similarity(image_embeddings: torch.Tensor, text_embeddings: torch.Tensor) -> torch.Tensor
```

Implementation requirements:

- return CPU tensors from `encode_text` and `encode_images`
- L2-normalize embeddings before returning
- support `device="cpu"` and `device="cuda"`
- support `device=None` auto-selection
- avoid changing existing CLIP behavior
- raise clear errors if model dependencies or model access fail

The model registry should keep a clear distinction between:

- implemented models
- registered placeholders
- unavailable models due to missing access or unsupported dependencies

## Testing Strategy

CI tests must remain lightweight and must not download model weights.

Recommended test layers:

1. Unit tests for registry metadata.
2. Unit tests for device resolution.
3. Unit tests using small fake wrapper classes or monkeypatching.
4. Optional local integration test script that is not run in CI.

Potential files:

- `tests/test_plip_registry.py`
- `tests/test_model_interface_contract.py`
- `examples/05_model_comparison_demo.py` later, only after wrappers are stable

Do not add tests that call `from_pretrained()` in CI.

## Optional Local Integration Test

If PLIP is implemented, provide a local-only smoke command:

```bash
python examples/02_zero_shot_classification_demo.py \
  --config configs/zero_shot_demo_config.json \
  --model plip \
  --device cuda
```

This should be documented as optional because it may download model weights and requires a suitable local environment.

## CLIP vs PLIP Comparison Protocol

Use the same sampled manifest and the same prompts for both models.

Recommended initial protocol:

1. Use `dataset/MHIST/manifest_test_50_per_class.csv`.
2. Run zero-shot classification with CLIP.
3. Run zero-shot classification with PLIP.
4. Save outputs under separate report directories:
   - `outputs/zero_shot_clip_mhist_sampled/`
   - `outputs/zero_shot_plip_mhist_sampled/`
5. Compare:
   - accuracy
   - balanced accuracy
   - macro-F1
   - per-class recall
   - confusion matrix
   - prediction distribution
   - examples in `errors.csv`

Do not claim PLIP is clinically valid even if metrics improve.

## Prompt Strategy

Start with the same prompts used in the CLIP baseline:

- `a histopathology image of hyperplastic polyp`
- `a histopathology image of sessile serrated adenoma`

Then optionally compare with pathology-specific prompt variants from the prompt template library.

Prompt sensitivity should remain an evaluation dimension, not an afterthought.

## Config Additions

Potential v0.2.0 config additions:

- `configs/model_comparison_mhist_config.json`
- `configs/zero_shot_plip_demo_config.json`

These should remain templates and should not include real data.

## Documentation Updates

If PLIP is implemented, update:

- `README.md` model registry table
- `docs/project_positioning.md`
- `docs/small_dataset_quickstart.md`
- `docs/mhist_baseline_observation.md` with a follow-up comparison section
- release notes for v0.2.0

If PLIP is not implemented due to a blocker, document the blocker in a short note rather than leaving the status ambiguous.

## Suggested v0.2.0 Sequence

1. Research PLIP loading requirements.
2. Add a model interface contract test.
3. Refactor CLIP wrapper only if needed to share a common interface.
4. Implement PLIP wrapper behind the existing registry.
5. Add registry and interface tests without model download.
6. Add optional local PLIP smoke documentation.
7. Run sampled MHIST comparison locally.
8. Document CLIP vs PLIP baseline observations.

## Release Criteria

v0.2.0 should only be released when:

- existing CLIP workflows still pass
- CI remains lightweight
- PLIP status is unambiguous
- docs do not overclaim clinical capability
- MHIST comparison results are saved locally but not committed
- release notes clearly state what is implemented and what remains planned
