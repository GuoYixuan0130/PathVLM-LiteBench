# v0.8.0 Plan

This document outlines the next development milestone after the v0.7.0 patch-coordinate heatmap release.

v0.8.0 should harden the prompt-scored patch-coordinate heatmap workflow that now exists on main. The goal is to make the path from coordinate manifest plus text prompt to score CSV plus heatmap predictable, configurable, and clearly separated from whole-slide image processing.

## Current Status

- v0.7.0 is tagged and published as a GitHub Release.
- Main is now on development version `0.8.0.dev0`.
- v0.7.0 contains the artifact-only patch-coordinate heatmap workflow through `render-coordinate-heatmap`.
- Main also contains `score-coordinate-heatmap`, which loads a CLIP-style model, scores coordinate patches against a text prompt, and writes `scores.csv` plus `heatmap.png`.
- CI tests scoring logic with fake models and does not download model weights.
- CONCH remains optional and gated.

## Milestone Goal

v0.8.0 should answer:

- Can users run prompt-scored patch-coordinate heatmaps from either CLI arguments or a JSON config?
- Can the scoring path remain patch-level, local-first, and reproducible?
- Can docs make the difference between artifact-only rendering and model-backed scoring explicit?
- Can tests cover config parsing, runtime overrides, and output path behavior without real model downloads?

## Preferred Scope

Recommended v0.8.0 scope:

1. Add config support for `score-coordinate-heatmap`.
2. Add a committed example config for prompt-scored patch-coordinate heatmaps.
3. Validate prompt-scored heatmap configs through `validate-config`.
4. Document when the scoring command loads a model and may download weights.
5. Keep CI coverage model-free by using fake model wrappers in tests.
6. Preserve CLI argument overrides for config-driven runs.
7. Keep generated score CSVs, heatmaps, caches, reports, and datasets ignored.

## Non-Goals

v0.8.0 should not add:

- whole-slide image file reading
- slide tiling or tissue detection
- slide pyramid rendering
- clinical diagnostic claims
- required model downloads in CI
- required CONCH access
- Hugging Face token handling
- committed generated heatmaps, scores, model weights, caches, or real pathology images

## Candidate Work Items

Highest priority:

- define `patch_coordinate_heatmap_scoring` config fields
- support `score-coordinate-heatmap --config ...`
- allow command-line overrides such as `--prompt`, `--output-dir`, `--score-csv`, `--heatmap-output`, `--model`, `--device`, and `--max-images`
- add config validation output for prompt-scored heatmap configs
- add tests that verify config loading and override behavior without model downloads

Medium priority:

- add `docs/prompt_scored_coordinate_heatmap_workflow.md`
- update README with a short v0.8.0 development note once the workflow is stable
- add a small synthetic smoke recipe that reuses generated patch tiles but keeps outputs ignored
- add optional metadata output for prompt, model, device, patch count, and output paths

Lower priority:

- add multi-prompt scoring over the same coordinate manifest
- add cached image embeddings for repeated prompt scoring
- add comparison utilities for multiple prompt-scored heatmaps

## Design Constraints

- Treat heatmaps as patch-coordinate score visualizations only.
- Use existing model registry behavior and `--device auto`.
- Keep config fields explicit and serializable.
- Keep the artifact-only renderer usable without model imports or model downloads.
- Keep tests lightweight and independent of real pathology images.
- Keep public wording focused on patch-level evaluation and visualization.

## Release Criteria

v0.8.0 should only be released when:

- `score-coordinate-heatmap` supports config-driven and argument-driven runs
- example config validation passes locally and in CI
- tests cover config parsing and runtime overrides
- docs explain data layout, model-loading behavior, outputs, and limitations
- README release links are updated during release preparation
- CI passes on the release commit
