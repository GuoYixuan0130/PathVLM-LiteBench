# v0.4.0 Plan

This document outlines the next development milestone after the v0.3.1 reporting release.

The goal of v0.4.0 is to make a careful CONCH integration decision. The first step is a local-only feasibility check, not an immediate wrapper implementation. CONCH has access, license, package, and loading requirements that are different from the current CLIP-compatible PLIP path, so the project should verify those constraints before changing the implemented model registry status.

## Current Status

- CLIP and PLIP are implemented through the existing model registry.
- CONCH feasibility checks passed locally, and CONCH is now supported as an optional implemented wrapper.
- Report summaries and multi-run comparisons are available through saved CSV/JSON artifacts.
- CI remains lightweight and does not download model weights.
- Generated datasets, reports, and model artifacts remain ignored through `.gitignore`.

The next gap is model breadth: PathVLM-LiteBench can compare CLIP and PLIP, but it cannot yet evaluate CONCH under the same retrieval, zero-shot, and prompt sensitivity interfaces.

## External Status Check

As of May 6, 2026, public CONCH integration should be treated as gated and optional:

- The Hugging Face model page for `MahmoodLab/CONCH` describes CONCH as a gated model with access terms and a CC-BY-NC-ND-4.0 license.
- The model card states that access requires agreeing to terms of use and that personal email domains may be denied.
- The model card usage example installs the CONCH repository from GitHub and loads weights through `create_model_from_pretrained`.
- The GitHub repository documents a separate package-style installation flow, not a plain `transformers.AutoModel` loading path.

Primary references:

- Hugging Face model card: <https://huggingface.co/MahmoodLab/CONCH>
- CONCH GitHub repository: <https://github.com/mahmoodlab/CONCH>

These constraints make CONCH a poor candidate for a blind wrapper implementation. The correct next step is to document and run local feasibility checks first.

## Milestone Goal

v0.4.0 should answer:

- Can CONCH be loaded in a local development environment with the current project dependencies plus optional extras?
- Does the model expose stable image and text embedding methods that can match the existing wrapper contract?
- Can embeddings be normalized and returned as CPU tensors like CLIP and PLIP?
- Can the wrapper support CPU and CUDA without breaking lightweight CI?
- What authentication or license constraints must be documented?
- Should `conch` remain optional-only, or should additional local demo/config support be added around it?

## Preferred Outcome

The preferred v0.4.0 outcome is:

1. Add `docs/conch_feasibility_check.md`.
2. Run local-only access, loading, embedding, CPU, and CUDA checks.
3. Record exact package versions, model identifiers, checkpoint access path, and blockers.
4. Implement `CONCHWrapper` only if the checks pass cleanly.
5. Keep CI tests synthetic and offline even if the wrapper is implemented.

If access or dependency constraints are not clean in another environment, CONCH should fail with a clear setup error rather than becoming a required dependency.

## Non-Goals

v0.4.0 should not add:

- clinical diagnostic claims
- bundled CONCH weights
- automatic Hugging Face token handling beyond standard environment variables
- committed model outputs or generated comparison reports
- CONCH downloads in CI
- new dashboard or tracking infrastructure
- full WSI high-throughput processing
- model training or fine-tuning

## CONCH Feasibility Checklist

Before implementation, verify locally:

- exact Hugging Face model identifier and case sensitivity
- whether access to weights is approved for the local Hugging Face account
- license and permitted usage constraints
- package installation path for `mahmoodlab/CONCH`
- Python and PyTorch compatibility with this project
- CPU model loading behavior
- CUDA model loading behavior
- peak CUDA memory for a tiny image smoke test
- text embedding API
- image embedding API
- embedding dimensionality and dtype
- whether embeddings can be L2-normalized
- whether image and text embeddings share the same contrastive space
- whether preprocessing can be wrapped behind the existing image loader flow
- error messages when token/access is missing

Do not add CONCH downloads to CI even though local feasibility checks passed.

## Wrapper Design Principles

If implemented, `CONCHWrapper` should match the current model interface:

```python
encode_text(texts: list[str]) -> torch.Tensor
encode_images(images: list[Image.Image]) -> torch.Tensor
compute_similarity(image_embeddings: torch.Tensor, text_embeddings: torch.Tensor) -> torch.Tensor
```

Implementation requirements:

- return CPU tensors from `encode_text` and `encode_images`
- L2-normalize returned embeddings when used for retrieval and zero-shot classification
- support `device="cpu"` and `device="cuda"`
- support `device=None` auto-selection
- avoid changing CLIP or PLIP behavior
- fail clearly when CONCH dependencies, weights, or authentication are unavailable
- keep CONCH dependencies optional unless there is a strong reason to promote them to core dependencies

## Registry Rules

The model registry should keep statuses explicit:

- `clip`: implemented baseline
- `plip`: implemented CLIP-compatible pathology model
- `conch`: optional implemented wrapper that requires gated Hugging Face access and the CONCH package

CONCH implementation updates include:

- registry metadata
- CLI `models` output expectations
- `tests/test_model_registry.py`
- README model registry table
- `docs/project_positioning.md`
- release notes

If CONCH remains blocked, update the registry description only if needed and document the blocker in `docs/conch_feasibility_check.md`.

## Testing Strategy

CI tests must remain lightweight and offline.

Recommended tests:

- registry metadata tests
- dependency-missing error tests using monkeypatching
- wrapper interface contract tests with fake model objects
- device-resolution tests reused from existing patterns
- no tests that call `from_pretrained()` or download weights in CI

Optional local checks should remain documented commands, not CI jobs.

## Optional Local Smoke Protocol

After access is approved and dependencies are installed, run a tiny local smoke test:

```bash
python -c "import torch; print(torch.__version__); print(torch.cuda.is_available())"
```

Then verify CONCH package import and model loading with the official repository's recommended loading path. Record the exact commands, outputs, and errors in `docs/conch_feasibility_check.md`.

If a wrapper is implemented, run existing demos with `--model conch` only as optional local checks:

```bash
python examples/01_patch_text_retrieval_demo.py --model conch --device cpu
python examples/02_zero_shot_classification_demo.py --model conch --device cpu
python examples/03_prompt_sensitivity_demo.py --model conch --device cpu
```

CUDA checks should be optional and clearly marked as hardware-dependent.

1. Add v0.4.0 plan. Done.
2. Add `docs/conch_feasibility_check.md` with commands and result log template.
3. Run local metadata/access checks. Done.
4. Run local CPU text and image embedding checks if access is available. Done.
5. Run local CUDA smoke checks if hardware is available. Done.
6. Decide whether implementation is low-risk. Done.
7. Implement `CONCHWrapper` only after the feasibility result is clear. Done.
8. Add offline tests and documentation. Done.
9. Run CLIP vs PLIP vs CONCH comparison only as a local optional experiment.

## Release Criteria

v0.4.0 should only be released when:

- CONCH status is unambiguous
- CI remains lightweight and offline
- existing CLIP and PLIP workflows still pass tests
- docs clearly state authentication and license constraints
- generated outputs and model weights are not committed
- no clinical diagnostic claims are introduced
- release notes explain that CONCH is optional and requires gated Hugging Face access
