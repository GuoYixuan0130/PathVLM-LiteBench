# v0.10.0 Plan

This document outlines the next development milestone after the v0.9.0 prompt-scored patch-coordinate heatmap reproducibility release.

v0.10.0 should make repeated prompt-scored patch-coordinate heatmap experiments easier to run as a coherent prompt set. The goal is to let users define several prompt runs over the same coordinate manifest, produce consistent per-prompt artifacts, and compare the saved outputs without turning PathVLM-LiteBench into a whole-slide image processing framework.

## Current Status

- v0.9.0 is tagged and published as a GitHub Release.
- Main is now on development version `0.10.0.dev0`.
- v0.9.0 contains the prompt-scored coordinate heatmap reproducibility layer:
  - `score-coordinate-heatmap` metadata sidecar output
  - `score-coordinate-heatmap --dry-run`
  - `metadata_output` config support
  - artifact-only `compare-coordinate-heatmap-scores`
  - comparison CSV and optional Markdown summary output
- Prompt-set coordinate heatmap configs are now loadable and validatable through `validate-config`.
- `score-coordinate-heatmap-prompt-set --dry-run` expands prompt-set run names and output paths without loading images or models.
- `score-coordinate-heatmap-prompt-set` can run prompt-set scoring and save per-prompt `scores.csv`, `heatmap.png`, and `metadata.json` artifacts.
- Prompt-set scoring now writes artifact-only comparison CSV and Markdown summaries from saved per-prompt score artifacts.
- v0.7.0 artifact-only heatmap rendering remains available through `render-coordinate-heatmap`.
- v0.8.0 model-backed single-prompt scoring remains available through `score-coordinate-heatmap`.
- CI tests scoring logic with fake model wrappers and does not download model weights.
- CONCH remains optional and gated.

## Milestone Goal

v0.10.0 should answer:

- Can users define a prompt set once and run consistent prompt-scored patch-coordinate heatmap outputs?
- Can the same coordinate manifest and output conventions be reused across several prompt runs?
- Can users inspect and compare a prompt set from saved artifacts without rerunning model inference?
- Can batch-style prompt experiments remain local-first, model-download aware, and CI-friendly?
- Can the workflow remain patch-level without adding slide readers, tissue detection, or clinical interpretation?

## Preferred Scope

Recommended v0.10.0 scope:

1. Add a prompt-set config for repeated prompt-scored coordinate heatmap runs.
2. Add a CLI entry point for running a coordinate heatmap prompt set from config.
3. Add a dry-run path that expands prompt-set runs and output paths without loading a model.
4. Save per-prompt `scores.csv`, `heatmap.png`, and `metadata.json` artifacts under predictable run directories.
5. Produce or reuse an artifact-only comparison summary for the completed prompt set.
6. Document prompt-set output layout, dry-run behavior, model-loading behavior, and artifact hygiene.
7. Keep generated outputs, score CSVs, heatmaps, metadata files, comparison files, datasets, caches, and weights ignored.

## Non-Goals

v0.10.0 should not add:

- whole-slide image file reading
- slide tiling or tissue detection
- slide pyramid rendering
- clinical diagnostic claims
- tumor boundary identification claims
- required model downloads in CI
- required CONCH access
- Hugging Face token handling
- bundled real pathology images or whole-slide image files
- committed generated heatmaps, score CSVs, metadata outputs, comparison artifacts, reports, caches, model weights, or local datasets

## Candidate Work Items

Completed first slice:

- design a JSON config for prompt-set coordinate heatmap experiments
- include shared fields such as manifest path, output root, model, device, image root, coordinate columns, colormap defaults, and max-images smoke settings
- include prompt entries with stable keys, prompt text, optional titles, optional output subdirectories, and optional colormap overrides
- add config loading, validation, serialization, and example config coverage
- add a prompt-set CLI dry-run mode that prints expanded run names and output paths without loading images or models
- add tests proving prompt-set dry-run does not create a model
- run prompt-set scoring through existing single-prompt scoring helpers where possible
- save per-prompt metadata that records prompt key, prompt text, shared manifest, model, device, patch count, and artifact paths
- add focused fake-model tests for prompt-set execution without downloading weights
- generate an artifact-only comparison CSV or Markdown summary after successful prompt-set runs

Highest priority:

- document how prompt-set outputs can be compared with `compare-coordinate-heatmap-scores`

Medium priority:

- document prompt-set dry-run behavior and output layout

Lower priority:

- add artifact-only per-patch score delta utilities for two saved prompt runs
- add cached image embeddings for repeated prompts over the same coordinate manifest
- add multi-model prompt-set comparisons
- add optional richer comparison statistics such as correlation or score rank overlap from saved artifacts

## Design Constraints

- Treat heatmaps as patch-coordinate score visualizations only.
- Keep `render-coordinate-heatmap` artifact-only and model-free.
- Keep `compare-coordinate-heatmap-scores` artifact-only and model-free.
- Keep prompt-set dry-run model-free and image-loading-free.
- Use existing model registry behavior and `--device auto` for model-backed scoring.
- Keep configs explicit and serializable.
- Prefer composing existing scoring and comparison helpers over duplicating model or artifact logic.
- Keep tests lightweight and independent of real pathology images.
- Keep public wording focused on patch-level evaluation and visualization.

## Release Criteria

v0.10.0 should only be released when:

- prompt-set config validation is covered by tests
- prompt-set dry-run is covered by tests and does not load models
- prompt-set execution is covered with fake model wrappers and no model downloads
- generated prompt-set artifacts are documented and ignored
- artifact-only comparison behavior remains model-free
- docs explain output layout, model-loading behavior, artifact-only comparison behavior, and limitations
- README release links are updated during release preparation
- CI passes on the release commit and tag
