# v0.5.0 Plan

This document outlines the next development milestone after the v0.4.1 prompt-grid release.

The recommended v0.5.0 direction is a packaged public patch-dataset benchmark workflow. The goal is to make PathVLM-LiteBench easier to run on a reproducible non-toy pathology dataset while keeping the project low-compute, patch-level, and local-first.

## Current Status

- CLIP, PLIP, and optional CONCH are available through the model registry.
- Zero-shot classification, retrieval, prompt sensitivity, report summaries, report comparisons, and zero-shot prompt grids are implemented.
- MHIST local observations document class bias and prompt sensitivity across CLIP, PLIP, and CONCH.
- CI remains lightweight and offline.
- Local datasets, generated reports, prompt-grid outputs, demo patches, embeddings, and model artifacts remain ignored through `.gitignore`.

## Milestone Goal

v0.5.0 should turn the existing local workflows into a clearer benchmark path for a public patch-level pathology dataset.

The milestone should answer:

- What is the simplest public patch dataset workflow a new user can reproduce?
- Can users go from downloaded annotations to manifest, sampled manifest, zero-shot run, prompt-grid run, and comparison report with documented commands?
- Can the workflow stay laptop-friendly and avoid committed data or generated outputs?
- Can the docs explain how to interpret accuracy, balanced accuracy, macro-F1, predicted-label distributions, confusion matrices, and prompt-grid warnings without making clinical claims?

## Preferred Scope

Recommended v0.5.0 scope:

1. Add a public patch-dataset workflow guide.
2. Add one or two example benchmark configs for sampled runs.
3. Add a prompt-grid workflow guide that explains config fields and output structure.
4. Add checks or tests for the example configs where possible without requiring data downloads.
5. Keep all real data and generated reports local and ignored by Git.

The MHIST workflow is the most practical first target because the repository already has conversion, sampling, and observation docs around it. If another public patch-level dataset is chosen later, it should still use the same standard manifest format.

## Non-Goals

v0.5.0 should not add:

- clinical diagnostic claims
- full WSI high-throughput processing as a required workflow
- raw WSI parsing dependencies in the core package
- model training or fine-tuning
- bundled datasets, model weights, or generated reports
- automatic Hugging Face token handling
- hosted dashboards or experiment tracking

Optional sampled WSI heatmaps can remain a later direction, but they should not displace the patch-level benchmark workflow unless the project has a clear lightweight path.

## Candidate Work Items

Highest priority:

- `docs/prompt_grid_workflow.md`: explain `run-zero-shot-grid`, config fields, output layout, dry-run mode, and interpretation.
- `docs/public_patch_benchmark_workflow.md`: a step-by-step local workflow from dataset preparation to saved comparison.
- example config for a sampled MHIST zero-shot prompt grid, already started with `configs/zero_shot_prompt_grid_mhist_sample.json`.
- config validation tests or smoke tests that ensure example configs parse without loading models.

Medium priority:

- a small helper command that validates prompt-grid configs without running inference.
- a workflow summary document that links conversion, sampling, zero-shot, prompt-grid, summarize, and compare commands.
- documentation for naming conventions under `outputs/` so generated reports are easy to compare and ignore.

Lower priority:

- more prompt variants for MHIST.
- larger local prompt-grid observations.
- optional WSI-level text-guided heatmap prototype.

## Release Criteria

v0.5.0 should only be released when:

- at least one public patch-dataset workflow is documented end to end
- prompt-grid config usage is documented beyond the README snippet
- example configs are tested or manually verified in dry-run mode
- CI remains offline and lightweight
- generated datasets, outputs, and model artifacts are not committed
- documentation keeps results framed as local model-behavior observations, not clinical interpretation

## Suggested First Step

The next implementation step should be `docs/prompt_grid_workflow.md`.

That document is small, directly supports the new v0.4.1 CLI, and gives users enough context to run prompt grids without reading source code. After that, `docs/public_patch_benchmark_workflow.md` can connect manifest conversion, sampling, prompt grids, and report comparison into one benchmark workflow.
