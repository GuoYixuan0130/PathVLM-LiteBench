# v0.7.0 Plan

This document outlines a conservative next milestone after the v0.6.0 reproducible MHIST benchmark audit release.

The recommended v0.7.0 direction is an optional WSI-oriented, text-guided heatmap prototype that keeps PathVLM-LiteBench focused on patch-level vision-language model evaluation. The milestone should make it easier to inspect patch-level VLM scores across slide-derived patch coordinates without turning the project into a full WSI processing framework.

## Current Status

- v0.6.0 is tagged and published as a GitHub Release.
- The v0.6.0 post-release audit confirms release, tag, README, version, CLI, config validation, dry-run, and CI state.
- CLIP, PLIP, and optional CONCH are available through the model registry.
- Public patch benchmark workflows are documented and validated with sampled MHIST configs.
- `run-zero-shot-grid` supports runtime output overrides for non-overwriting local runs.
- Generated datasets, outputs, demo patches, model weights, embeddings, Python build metadata, and benchmark reports remain ignored.
- README still lists an optional WSI-level text-guided heatmap demo as an open roadmap item.
- Coordinate-aware patch manifest loading and offline heatmap utilities have started the WSI-oriented milestone without adding WSI-reader dependencies.

## Milestone Goal

v0.7.0 should answer:

- Can users visualize text-guided model scores over pre-extracted slide patches with coordinates?
- Can the workflow remain patch-level and avoid requiring WSI readers such as OpenSlide or TIAToolbox?
- Can the feature reuse existing frozen model wrappers, embedding utilities, and report hygiene?
- Can the documentation clearly separate WSI-oriented visualization from clinical interpretation?

## Preferred Scope

Recommended v0.7.0 scope:

1. Define a coordinate-aware patch manifest extension for slide-derived patches.
2. Add a lightweight heatmap utility that consumes patch paths, coordinates, and text scores.
3. Provide a demo or CLI path that works with pre-extracted patches only.
4. Save heatmap visualizations and structured score artifacts under `outputs/`.
5. Add config validation for the new heatmap workflow if a config-driven entry point is introduced.
6. Add offline tests for coordinate parsing, score-to-grid aggregation, and output path handling.
7. Document the workflow with synthetic or user-provided patch coordinates only.

## Non-Goals

v0.7.0 should not add:

- bundled WSI files or real pathology images
- required OpenSlide, TIAToolbox, cuCIM, or other heavy WSI dependencies
- required model downloads in CI
- slide tiling, tissue detection, stain normalization, or WSI pyramid rendering
- clinical diagnostic claims
- automatic Hugging Face token handling
- required CONCH access
- committed generated heatmaps, predictions, run logs, or local dataset metadata

The milestone should remain compatible with CPU-only CI and local laptop workflows.

## Candidate Work Items

Highest priority:

- specify accepted coordinate columns for manifest rows, such as `x`, `y`, optional `width`, `height`, and optional `slide_id`
- add a small parser or validation helper for coordinate-aware manifests
- add a score-grid aggregation utility that can be tested without loading models
- add a minimal Matplotlib heatmap renderer for precomputed patch scores
- document an end-to-end workflow using pre-extracted patches and ignored local outputs

Medium priority:

- add a config example for text-guided heatmap generation
- support multiple text prompts in one heatmap run
- add summary metadata that records prompt text, model key, device, patch count, and coordinate bounds
- include a synthetic demo manifest that contains no real pathology data

Lower priority:

- add optional overlay support when users provide a downsampled slide thumbnail
- add cached embeddings for repeated prompt scoring over the same patch set
- add comparison utilities for heatmaps from multiple prompts or models

## Design Constraints

- Use existing model registry keys and `--device auto` behavior.
- Keep model inference optional in tests by isolating parsing, aggregation, and rendering logic.
- Write generated artifacts only under ignored output directories.
- Prefer explicit config fields over implicit path conventions.
- Keep CONCH optional and gated.
- Treat WSI-oriented visualization as a patch-coordinate visualization workflow, not as full WSI support.

## Release Criteria

v0.7.0 should only be released when:

- the heatmap workflow can run on pre-extracted patches with coordinate metadata
- offline tests cover coordinate parsing and heatmap artifact generation
- docs explain required local data layout and output hygiene
- CI remains lightweight and does not require model downloads or real pathology data
- README current release and roadmap links are updated during the release preparation step
- results are framed as text-guided patch score visualizations, not clinical findings
